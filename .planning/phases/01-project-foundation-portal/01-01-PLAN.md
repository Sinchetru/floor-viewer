1. Create `src/middleware.ts` with cookie-based route protection:
```typescript
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  const payloadToken = request.cookies.get('payload-token')

  if (!payloadToken) {
    return NextResponse.redirect(new URL('/login', request.url))
  }

  return NextResponse.next()
}

export const config = {
  matcher: ['/portal/:path*', '/flaechen/:path*'],
}
```
CRITICAL: The matcher must NOT include `/login` or `/api` — that causes infinite redirect loops. Only protect `/portal` and `/flaechen` paths.

CRITICAL: File location is `src/middleware.ts` (inside `src/`, not inside a route group). Next.js picks up middleware from project root or `src/` root.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Middleware | Cookie presence checked before accessing protected routes |
| Middleware → Server Component | Middleware is a fast check only; Server Components must re-verify with payload.auth() |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Elevation of privilege | src/middleware.ts | mitigate | Middleware checks cookie presence; portal Server Component (Plan 03) calls payload.auth() for authoritative verification — defense in depth per CVE-2025-29927 |
| T-01-02 | Spoofing | src/middleware.ts | mitigate | Only check for `payload-token` cookie name (Payload's official cookie); do not trust custom headers |
| T-01-03 | Denial of service | Middleware redirect loop | mitigate | Matcher explicitly excludes /login and /api paths — no redirect loop possible |
</threat_model>
