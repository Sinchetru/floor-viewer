# Plan 02-01: Define RoomGeometry and RoomData Collections

## Goal

Create `src/collections/RoomGeometry.ts` and `src/collections/RoomData.ts`, register both in `src/payload.config.ts`, and regenerate TypeScript types. All collection operations are admin-only.

## Context

Phase 2 delivers the two Payload collections that every downstream phase depends on. No map rendering, no KST visibility filtering — this plan defines the schema only.

Depends on: Phase 1 complete (Users collection, admin access helper exist).

**Existing patterns to follow:**
- Collection definition: `src/collections/Users.ts` (fields array, access object)
- Admin-only access helper: `src/collections/access/admin.ts` exports `admin: Access`
- Registration: `src/payload.config.ts` `collections` array

**Data schema (authoritative from PROJECT.md):**

RoomGeometry: `room_id` (text, unique), `path_data` (text), `building` (text), `floor` (text)

RoomData: `room_id` (text, unique), `room_number`, `room_type`, `room_type_code`, `area_sqm` (number), `din277_code`, `din277_name`, `cost_center`, `cost_center_name`, `special_use`, `owner`, `location`, `room_note`, `usage_note`, `usage_from` (date), `usage_to` (date)

<threat_model>
## Threat Model (ASVS L1)

| Threat | Severity | Mitigation |
|--------|----------|------------|
| Unauthorized read of room geometry/data by non-admin | High | `read: admin` on both collections — only admin role can query via Payload |
| Unauthorized create/update/delete by non-admin | High | `create`, `update`, `delete` all set to `admin` access helper |
| Payload admin panel exposes collection to non-admin users | High | `access.admin` inherited from Users collection pattern; Payload panel already restricted to admin role |
| Schema drift — room_id not unique, allows duplicate rows | Medium | Payload `unique: true` constraint on room_id field prevents duplicates at DB level |
</threat_model>

## Tasks

### Task 1: Create RoomGeometry collection

**File:** `src/collections/RoomGeometry.ts`

**Action:** Create the collection following the `Users.ts` pattern. Import `admin` from `./access/admin`. Set all access operations (`read`, `create`, `update`, `delete`) to `admin`. No `auth: true` (these are data collections, not user collections).

Field definitions:
```typescript
fields: [
  {
    name: 'room_id',
    type: 'text',
    required: true,
    unique: true,
    admin: { description: 'Join key — Format: 16.11.00.018' },
  },
  {
    name: 'path_data',
    type: 'textarea',
    required: true,
    admin: { description: 'SVG d attribute string for the room polygon' },
  },
  {
    name: 'building',
    type: 'text',
    required: true,
    admin: { description: 'Building code — e.g. 1611' },
  },
  {
    name: 'floor',
    type: 'text',
    required: true,
    admin: { description: 'Floor identifier — e.g. 00, 01, U1' },
  },
]
```

Set `admin.useAsTitle: 'room_id'` and `slug: 'room-geometry'`.

**Details:** Do not add any extra fields beyond the schema. `path_data` uses `textarea` type (not `text`) because SVG path strings can be very long.

### Task 2: Create RoomData collection

**File:** `src/collections/RoomData.ts`

**Action:** Same pattern as RoomGeometry. Import `admin` from `./access/admin`. All access set to `admin`. `slug: 'room-data'`, `admin.useAsTitle: 'room_id'`.

Field definitions:
```typescript
fields: [
  { name: 'room_id', type: 'text', required: true, unique: true },
  { name: 'room_number', type: 'text' },
  { name: 'room_type', type: 'text' },
  { name: 'room_type_code', type: 'text' },
  { name: 'area_sqm', type: 'number' },
  { name: 'din277_code', type: 'text' },
  { name: 'din277_name', type: 'text' },
  { name: 'cost_center', type: 'text', admin: { description: 'Kostenstelle — used for visibility scoping' } },
  { name: 'cost_center_name', type: 'text' },
  { name: 'special_use', type: 'text' },
  { name: 'owner', type: 'text' },
  { name: 'location', type: 'text' },
  { name: 'room_note', type: 'text' },
  { name: 'usage_note', type: 'text' },
  { name: 'usage_from', type: 'date' },
  { name: 'usage_to', type: 'date' },
]
```

**Details:** `area_sqm` is `number` type. `usage_from` and `usage_to` are `date` type. All other fields are `text`. None of the non-`room_id` fields are `required` (CSV import may have partial data).

---

### Task 3: Register collections + run schema migration [BLOCKING]

**Files:** `src/payload.config.ts`

**Action:**

1. Add imports to `payload.config.ts`:
   ```typescript
   import { RoomGeometry } from './collections/RoomGeometry'
   import { RoomData } from './collections/RoomData'
   ```
2. Add to `collections` array: `collections: [Users, Media, RoomGeometry, RoomData]`
3. Run type generation: `pnpm payload generate:types`
4. Run schema migration: `CI=true npx payload migrate`

**Details:** Type generation must run after both collections are registered. The migration must complete before any subsequent plan can insert data. If the migration prompts interactively, run with `CI=true PAYLOAD_MIGRATING=true npx payload migrate` to suppress prompts.
