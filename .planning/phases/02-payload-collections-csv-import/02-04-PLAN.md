# Plan 02-04: Seed CSV Files and Dev Seed Script

## Goal

Create two hand-crafted seed CSV files (`data/seed-geometry.csv` and `data/seed-roomdata.csv`) and a seed script (`scripts/seed.ts`) that reads them and upserts the data into Payload. These files are placeholders designed to be replaced with real exports from the facility management system.

## Context

Depends on: Plan 02-01 (collections exist, migration run), Plan 02-02 (import logic exists in `src/lib/csv-import.ts` for reuse).

**D-01/D-02:** `seed-geometry.csv` — hand-crafted rows with dummy SVG paths, realistic room_id format, correct building/floor values. A few rooms only — enough to test map rendering later.

**D-03:** `seed-roomdata.csv` — small slice of realistic room data with German column headers, matching the room_ids in the geometry file.

**D-04:** Seed script reads these CSVs and upserts via Payload local API. No hardcoded room data in the script itself.

**D-05:** Do NOT process any files from `D:/Projects/do_yourself`. Seed files are hand-crafted from scratch.

**room_id format from real data:** `16.11.00.018` (building=1611, subbuilding=11, floor=00, number=018)

The seed script runs as a standalone Node script using tsx (already available via Payload's dev toolchain). It initializes Payload in local API mode (no HTTP server needed).

### Task 2: Create seed script

**File:** `scripts/seed.ts`

**Action:** Create a TypeScript seed script that:

1. Guards against production: `if (process.env.NODE_ENV === 'production') { console.error('Seed must not run in production'); process.exit(1) }`
2. Reads `data/seed-geometry.csv` and `data/seed-roomdata.csv` using Node's `fs.readFileSync` (not the import routes — direct file reads for a CLI script).
3. Imports `parseAndMapCsv`, `GEOMETRY_HEADER_MAP`, `ROOMDATA_HEADER_MAP` from `../src/lib/csv-import`.
4. Initializes Payload local API:
   ```typescript
   import { getPayload } from 'payload'
   import config from '../src/payload.config'
   const payload = await getPayload({ config })
   ```
5. For geometry rows: for each row, upsert into `room-geometry` collection (same find-then-create-or-update logic as the route handler in Plan 02-02 Task 3).
6. For room data rows: parse `area_sqm` to float, parse date fields if present, upsert into `room-data` collection.
7. Log a summary to stdout: `console.log('Geometry: imported X, skipped Y')` and `console.log('Room data: imported X, skipped Y')`.
8. Call `process.exit(0)` at the end (Payload's local API keeps the event loop alive otherwise).

**Details:** Import paths use `../src/` relative paths since the script is in `scripts/`. The script is run with `npx tsx scripts/seed.ts` (tsx is already installed as a Payload dev dependency — verify with `pnpm list tsx`; if not present, add with `pnpm add -D tsx`).

Add a `seed` script to `package.json` scripts: `"seed": "npx tsx scripts/seed.ts"`

### Task 3: Verify end-to-end import

**Action:** Run the seed script against the dev database and confirm rows appear in Payload admin.

Steps:
1. Ensure dev server is NOT running (seed uses local API, not HTTP)

3. Confirm output shows 5 geometry rows imported, 5 room data rows imported, 0 skipped
4. Start dev server: `pnpm dev`
5. Log in as admin at `http://localhost:3000/admin`

7. Navigate to Room Data collection — 5 rows visible
8. Run seed script a second time — verify it upserts (row counts stay at 5, no duplicates created)
