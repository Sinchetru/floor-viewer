# Plan 02-03: Payload Admin Import UI Components

## Goal

Wire a CSV upload button into the Payload admin panel for each collection (RoomGeometry and RoomData) using Payload 3.x Custom Components. Admins can upload a CSV directly from the collection list view.

## Context

Depends on: Plan 02-01 (collections exist), Plan 02-02 (import routes exist at `/api/import/room-geometry` and `/api/import/room-data`).

**D-16:** Each collection gets an upload button in the Payload admin panel — implemented as a Payload Custom Component.

**D-17:** Claude's discretion — exact component pattern. Using Payload 3.x `admin.components.beforeList` injection point (renders a component above the collection list table) with a React client component containing a file input and fetch call. This approach:
- Does not require a custom view (which would replace the entire list view)
- Appears naturally above the existing collection list
- Works with Payload 3.x's import map system

**Payload 3.x Custom Component registration (required):**
Components used in `admin.components` must be registered in `payload.config.ts`'s `admin.importMap.baseDir`. The component file path is passed as a string reference in the collection config.

**Payload 3.x `beforeList` injection:**
```typescript
admin: {
  components: {
    beforeList: ['/path/to/Component#ExportName'],
  }
}
```

## Threat Model (ASVS L1)

| Threat | Severity | Mitigation |
|--------|----------|------------|
| Client-side upload component bypasses auth (CSRF-style direct POST) | High | The route handler (Plan 02-02) re-validates admin session server-side. Component sending the request is inside the Payload admin panel — Payload's session cookie is sent automatically with same-origin fetch |
| Component renders for non-admin users | Medium | Payload admin panel itself is restricted to admin role (wired in Phase 1). The `beforeList` component only appears inside `/admin` — non-admins never reach this UI |
| Upload feedback misleads admin about import status | Low | Component displays the full summary JSON response (total, imported, skipped, errors) so the admin can confirm what was actually written |
</threat_model>

## Tasks

### Task 1: Create shared upload component

**File:** `src/components/admin/CsvImportButton.tsx`

**Action:** Create a React client component (`'use client'`) that renders a file upload button. The component accepts a prop `endpoint: string` (the API route path) and `label: string` (button label for the collection).

Behavior:
1. Renders a `<div>` containing a `<label>` styled as a button (Tailwind: `cursor-pointer inline-flex items-center gap-2 px-4 py-2 bg-blue-600 text-white rounded hover:bg-blue-700 text-sm font-medium`) wrapping a hidden `<input type="file" accept=".csv,text/csv">`.
2. On file selection (`onChange`): set `loading: true`, build a `FormData` with `formData.append('file', file)`, POST to `endpoint` with `fetch(endpoint, { method: 'POST', body: formData })`.
3. On response: parse JSON, set `result` state, set `loading: false`.
4. Render result summary below the button:
   - If `result.imported > 0`: green text "Importiert: {result.imported} | Übersprungen: {result.skipped}"
   - If `result.errors?.length > 0`: show each error as a red list item: "Zeile {error.row}: {error.reason}"
   - If `result.error`: show red text with the error message
5. Reset file input after each upload so the same file can be re-uploaded.

Wrap the entire component in a `<div className="mb-4 p-4 border border-gray-200 rounded bg-gray-50">` so it is visually separated from the list table.

**Details:** Import types from `src/payload-types.ts` are not needed here — this is a pure UI component. No Payload SDK imports in this file (it runs in the browser).

### Task 2: Create collection-specific wrapper components

**Files:**
- `src/components/admin/RoomGeometryImport.tsx`
- `src/components/admin/RoomDataImport.tsx`

**Action:** Create two thin wrapper components. Each imports `CsvImportButton` and passes the correct props.

`RoomGeometryImport.tsx`:
```tsx
import CsvImportButton from './CsvImportButton'

export default function RoomGeometryImport() {
  return (
    <CsvImportButton
      endpoint="/api/import/room-geometry"
      label="Raumgeometrie CSV importieren"
    />
  )
}
```

`RoomDataImport.tsx`:
```tsx
import CsvImportButton from './CsvImportButton'

export default function RoomDataImport() {
  return (
    <CsvImportButton
      endpoint="/api/import/room-data"
      label="Raumdaten CSV importieren"
    />
  )
}
```

**Details:** These wrappers exist because Payload's `beforeList` references a specific file path + export name. Having separate files makes the collection config cleaner and the component reference unambiguous.

### Task 3: Register components in collection configs

**Files:**
- `src/collections/RoomGeometry.ts`
- `src/collections/RoomData.ts`

**Action:** Add `admin.components.beforeList` to each collection:

In `RoomGeometry.ts`, update the `admin` key:
```typescript
admin: {
  useAsTitle: 'room_id',
  components: {
    beforeList: ['@/components/admin/RoomGeometryImport#default'],
  },
},
```

In `RoomData.ts`, update the `admin` key:
```typescript
admin: {
  useAsTitle: 'room_id',
  components: {
    beforeList: ['@/components/admin/RoomDataImport#default'],
  },
},
```

**Details:** Payload 3.x resolves `@/` against the `admin.importMap.baseDir` set in `payload.config.ts` (which is `path.resolve(dirname)` = `src/`). The `#default` suffix specifies the default export. After editing, run `pnpm payload generate:importmap` to update the import map, then `pnpm build` to verify no errors.
